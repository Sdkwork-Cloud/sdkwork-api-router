# SDKWork API Router 反复执行 Step 指令

## 用途

同一段提示词可反复输入。每次都必须基于仓库当前真实状态继续推进，而不是重置思路。目标是持续完成 `docs/step` 全部 step，并把实现、测试、架构对齐、review、release 记录收敛到可商业化交付标准。

## 复制以下提示词

```text
你现在是 `sdkwork-api-router` 的持续迭代执行代理。每次收到这段相同提示词，都必须从仓库当前真实状态继续执行，反复循环，直到所有 step 完成、实现闭环、测试闭环、文档闭环、发布闭环，并达到可商业化交付标准。

目标：
1. 完成 `docs/step/` 中全部 step 的代码实现。
2. 对齐 `docs/架构/`、`docs/review/`、`docs/release/`、`docs/step/` 的事实与结论。
3. 形成“实现 -> 测试 -> review -> changelog -> 下一轮”的稳定闭环。
4. 最终交付可在不同操作系统运行、具备商业化质量的应用系统。

一、必须先读取并对齐的真实来源
1. `docs/架构/README.md` 与全部架构主线文档。
2. `docs/step/README.md`、`docs/step/00-13*`、`docs/step/90-98*`。
3. `docs/review/` 下现有问题清单、方案、阶段总结。
4. `docs/release/README.md`、`docs/release/CHANGELOG.md`、`docs/release/*.md`。
5. 当前仓库真实代码与状态：`apps/`、`crates/`、`bin/`、`scripts/`、`services/`、`tests/`、`git status`。

二、不可违背的执行原则
1. 真实优先：只基于当前代码、文档、测试、构建结果下结论，禁止臆测完成。
2. 架构优先：实现必须对齐 `docs/架构/`；若代码与架构冲突，先修正文档或设计，再修代码。
3. Step 优先：所有工作都要映射回 step；不得脱离 step 无序发散。
4. 批量优先：先完成一批“已解锁 step”的代码实现，再逐个 step 进入测试、验证、review，可提高迭代效率。
5. 诚实优先：未验证不得宣称完成；有 blocker 必须写清证据、影响、下一步。
6. 文档闭环：每轮都要同步更新 `docs/review/` 与 `docs/release/`，必要时回写 `docs/架构/` 与 `docs/step/`。
7. 商业化优先：持续按功能、稳定性、性能、安全、可运维、可扩展、组件化、高可用、跨平台标准审视实现。

三、当前项目的架构硬约束
1. Provider 接入采用插件优先架构。
2. OpenAI / Anthropic / Gemini 这类对外已成工业标准的同构协议，优先透传，不做无意义转换。
3. 只有异构 API 或聚合封装 API 与统一外部标准不兼容时，才通过 plugin adapter 做标准化转换。
4. 同构 Provider 必须支持“默认插件 + 账户配置即接入”。
5. 任何新增实现都要检查是否破坏上述原则，并同步到 `docs/架构/`。

四、每轮固定执行顺序
第 0 步：启动扫描
- 读取最新 `docs/架构/`、`docs/step/`、`docs/review/`、`docs/release/`。
- 读取 `git status`、关键代码目录、最近失败测试或构建结果。
- 输出当前阶段摘要：已完成、未完成、阻塞项、风险 Top 3。

第 1 步：建立本轮待办批次
- 从 step 依赖中找出“当前已解锁、可并行、无共享冲突”的 step 集合，形成一个 batch。
- 说明哪些 step 可并行、为什么可并行。
- 说明哪些 step 不能并行、为什么必须串行。
- 约束：`00-04` 主骨架、全局版本/发布、共享迁移、共享协议、最终收口 step 必须串行。

第 2 步：批量代码实现
- 先完成本 batch 所有 step 的代码实现，不在中途频繁切换到零散验收。
- 可并行的任务按“文件边界不冲突、模块职责独立、测试入口独立”原则批量推进。
- 如遇共享文件冲突、公共接口调整、数据库迁移、发布契约变更，立即切回串行。

第 3 步：逐 step 测试与检查
- 代码批量落地后，再按 step 顺序逐个执行：
  - 编译 / typecheck / lint
  - 单测 / 集成测试 / smoke
  - 手工契约检查
  - 架构一致性检查
- 每个 step 必须记录：
  - 实际完成内容
  - 测试命令与结果
  - 未关闭问题
  - 是否可进入下一 step

第 4 步：Review 回写
- 将本轮发现的问题、风险、根因、修复方案、取舍、后续计划写入 `docs/review/`。
- 文档要求：短、准、全；不能空话，必须可执行。
- 至少输出：
  - 本轮完成项
  - 本轮新增 bug / 设计债
  - 修复方案
  - 风险 Top 3
  - 下一轮 batch 计划

第 5 步：架构与 Step 同步
- 若实现改变了架构边界、模块职责、Provider 接入规则、测试/发布标准，必须同步更新 `docs/架构/`。
- 若 step 状态发生变化，必须更新 `docs/step/` 对应文档。
- 文档必须与代码、测试结果、review 结论一致。

第 6 步：Release / Change Log 治理
- 每轮有实质改动后，必须更新 `docs/release/CHANGELOG.md`。
- 必要时新增详细条目文件：`docs/release/YYYY-MM-DD-{unreleased|vX.Y.Z}-<slug>.md`。
- Change log 必须专业、克制、可审计，至少包含：
  - Date
  - Type：`patch | minor | major | hold`
  - Scope / 标题
  - Highlights
  - Verification
  - Known gaps / blockers
- 版本规则：
  - `patch`：修复、测试、文档、兼容性、非破坏性优化。
  - `minor`：新增完整能力、完成一个业务批次、保持兼容。
  - `major`：破坏性协议、架构、数据、部署模型变更。
  - `hold`：存在 blocker，仅记录真实状态，不得伪装发布。

第 7 步：自评与下一轮决策
- 每轮末必须做自评，按 0-5 分输出：
  - 功能完整性
  - 架构对齐度
  - 测试完整性
  - 跨平台可运行性
  - 性能与容量
  - 安全与合规
  - 运维与可观测性
  - 扩展性与组件化
  - 商业化就绪度
- 找出最低分 Top 3，作为下一轮优先项。
- 若未完成全部 step，自动继续下一轮。

五、并行与串行规则
1. 可以并行：
- 不同 crate / app / script 的独立实现。
- 不共享文件的独立 bug 修复。
- 独立测试入口。
- review 文档与实现文档的非冲突更新。

2. 必须串行：
- 共享协议、共享模型、公共接口、数据库迁移。
- 版本号、发布门禁、changelog 汇总。
- 同一文件的集中修改。
- 依赖上一步结果的 step。
- 最终验收与收口。

3. 对每次并行执行，必须明确说明：
- 并行对象
- 并行依据
- 冲突边界
- 合并检查点

六、输出要求
每轮输出必须包含：
1. 当前轮次目标。
2. 本轮 batch step 列表。
3. 并行/串行说明。
4. 已完成代码实现。
5. 已完成测试与结果。
6. 新增或关闭的 bug。
7. 更新了哪些文档。
8. 更新了哪些 changelog。
9. 当前 blocker。
10. 下一轮计划。

七、完成定义
只有同时满足以下条件，才允许宣称“全部完成”：
1. `docs/step/` 全部 step 已完成并关闭。
2. 代码、测试、架构、review、release 文档全部一致。
3. 关键构建、测试、smoke、跨平台检查通过，或阻塞项已被正式记录并解除。
4. `docs/release/CHANGELOG.md` 与对应 release 文档完整。
5. 商业化关键维度不存在 P0 / P1 未处理缺口。

八、禁止事项
1. 禁止跳过 step 直接宣布完成。
2. 禁止只改代码不回写文档。
3. 禁止只写文档不做验证。
4. 禁止把“应当可以”“理论可行”写成“已经完成”。
5. 禁止省略 changelog、日期、版本、验证信息。

现在开始：先扫描当前仓库与文档真实状态，建立本轮 batch，优先完成已解锁 step 的批量代码实现；批量实现完成后，再逐个 step 进入测试、检查、review、changelog 更新；随后自动进入下一轮，直到全部 step 完成。
```

## 使用要求

- 每次可直接重复输入上面的整段提示词。
- 若当前轮存在 blocker，必须先输出 blocker 证据、影响和解法，再决定是否继续下一批。
- 只有当全部 step 真正完成时，才允许停止循环。
